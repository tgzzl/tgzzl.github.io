前端高频知识点

Posted on 2019-12-23
  1. 防抖

触发高频事件后n秒内函数只会执行一次，如果n秒内高频事件再次被触发，则重新计算时间。在某段时间内，不管触发了多少次回调，只认最后一次

  • 思路： 每次触发事件时都取消之前的延时调用方法
  1. 节流

高频事件触发，但在n秒内只会执行一次，所以节流会稀释函数的执行频率。在某段时间内，不管你触发了多少次回调，只认第一次，并在计时结束时给予响应

  • 思路：每次触发事件时都判断当前是否有等待执行的延时函数
  1. 浏览器缓存

  • Memory Cache
  • Service Worker Cache
  • HTTP Cache
  • Push Cache (http2.0)

3.1 强缓存

命中强缓存的情况下，返回的 HTTP 状态码为 200

Response Headers version 优缺点
expires 1.0 浏览器对比本地时间和 expires 的时间戳，本地时间可修改，不可靠
cache-control 1.1 替代expires，更准确，优先级高

  • cache-control 的几种值比较

  • max-age=3600, s-maxage=31536000

    s-maxage仅在代理服务器(比如cache cdn)中生效，客户端中我们只考虑max-age

  • public 与 private

    是针对资源是否能够被代理服务缓存而存在的一组对立概念。private：默认值，该资源只能被浏览器缓存。public：既可以被浏览器缓存，也可以被代理服务器缓存

  • no-store 与 no-cache

    no-cache：绕开了浏览器直接向服务端去确认该资源是否过期

    no-store：不使用任何缓存策略

3.2 协商缓存

浏览器与服务器合作之下的缓存策略，浏览器需要向服务器去询问缓存的相关信息，从 Last-Modified 到 Etag

  • Last-Modified

    首次请求时响应头里返回Last-Modified时间戳，之后每次请求头里带上If-Modified-Since，其值为Last-Modified的值；服务器比对该时间戳和资源在服务器上的最后修改时间，如果未变化则响应304

    弊端：1. 编辑了文件但内容未改变，会被当做新资源，导致重新请求；2. 修改文件的速度在毫秒级时， If-Modified-Since检测不出来变化，导致请求了旧资源

  • Etag

    基于文件内容生成的唯一的标识字符串。首次请求时响应头里返回ETag，之后每次请求头里带上If-None-Match，其值为Etag的值；Etag 的生成过程需要服务器额外付出开销，在感知文件变化上比 Last-Modified 更加准确，优先级也更高，只能作为 Last-Modified 的补充和强化存在

3.3 HTTP 缓存决策指南

资源内容不可复用时，直接为 Cache-Control 设置 no-store，拒绝一切形式的缓存；否则考虑是否每次都需要向服务器进行缓存有效确认，如果需要，那么设 Cache-Control 的值为 no-cache；否则考虑该资源是否可以被代理服务器缓存，根据其结果决定是设置为 private 还是 public；然后考虑该资源的过期时间，设置对应的 max-age 和 s-maxage 值；最后，配置协商缓存需要用到的 Etag、Last-Modified 等参数。

4. let + for

变量i是let声明的，当前的i只在本轮循环有效，所以每一次循环的i其实都是一个新的变量，所以最后输出的是6。 JavaScript 引擎内部会记住上一轮循环的值，初始化本轮的变量i时，就在上一轮循环的基础上进行计算。

另外，for循环还有一个特别之处，就是设置循环变量的那部分是一个父作用域，而循环体内部是一个单独的子作用域。
